# 全基金 MVP 数据覆盖与训练准备设计

日期：2026-05-28

## 背景

当前系统已经有约 2.68 万只基金的本地搜索库，其中约 1.96 万只有东方财富排行口径的最新净值、涨跌幅和阶段收益。预测模型层仍以指数 ETF 为主，不能把所有基金都直接送入现有指数 ETF 模型，否则会把货币基金、债券基金、QDII、FOF、混合基金等不同资产行为混在一起，影响 MVP 可信度，也会污染后续高级模型训练数据。

本设计把“全基金覆盖”拆成三层：

1. 全库可搜索：所有基金都能被搜索、选择、进入预测入口。
2. 全库可解释：每只基金都返回明确的数据质量和模型覆盖状态。
3. 分层可训练：只有具备足够历史净值和合适标签的基金进入日频 MVP 模型；指数 ETF/LOF 保留指数增强模型。

## 目标

- 所有本地基金都能搜索到，预测接口不因模型未覆盖而失败。
- 修正涨跌幅来源边界：场内 ETF/LOF 优先实时行情，开放式基金使用最新净值或估值口径，货币基金使用收益口径，不混用。
- 新增全基金覆盖报告，量化基金总数、搜索覆盖、行情覆盖、历史样本覆盖、模型支持和回退原因。
- 新增开放式基金/LOF/货币基金历史净值采集入口，为后续高级模型训练准备基础样本。
- 保持前端展示诚实：模型支持、基线回退、暂不适合预测的基金类型必须明确可见。

## 非目标

- 本阶段不承诺所有 2.68 万只基金都完成历史全量抓取。
- 本阶段不把货币基金、债券基金、QDII、FOF 强行纳入指数 ETF 模型。
- 本阶段不声明达到可交易级准确率；高级模型训练前只交付可复现数据管线和覆盖验收。

## 推荐方案

采用“两层全基金 MVP”。

第一层是产品可用层：搜索、详情、预测入口覆盖所有基金。任何基金都能得到响应，但响应会携带模型覆盖状态和数据质量说明。

第二层是训练准备层：按基金类型和数据可得性构建历史样本。

- ETF/LOF/场内指数基金：继续使用现有指数增强数据管线，保留指数、期货、恐慌因子等特征。
- 开放式股票/混合/债券/指数联接基金：采集历史单位净值，构建通用日频净值动量、波动、回撤、阶段收益、标签样本。
- 货币基金：采集万份收益和七日年化，单独标注为收益型，不进入涨跌幅方向模型。
- QDII/FOF/数据不足基金：保留搜索和预测入口，默认进入显式回退，并写入覆盖报告。

## 后端设计

新增基金覆盖概念，不改变已有预测响应主结构：

- `model_supported`：当前基金和预测周期由模型服务完成推理。
- `baseline_only`：当前基金无可用模型，但可用后端基线给出非交易级估计。
- `unsupported_fund`：当前基金不适合现有模型，或历史样本不足。
- `model_unavailable`：模型服务不可用。

新增或扩展服务职责：

- 基金搜索继续读取本地 `funds.json`。
- 行情刷新按基金类型分流，避免用 ETF 实时价格覆盖开放式基金净值。
- 预测服务先判断基金类型和模型覆盖清单，再决定调用模型服务或回退。
- 生成覆盖报告 JSON，供验收脚本和前端调试使用。

## 训练数据设计

新增通用全基金日频数据集，与现有 `index_fund_daily_v1` 保持分离。

建议新增特征集：`all_fund_nav_daily_v1`。

基础列：

- `fund_code`
- `fund_name`
- `fund_type`
- `sample_trade_date`
- `asof_time`
- `nav`
- `accumulated_nav`
- `return_1d`
- `return_5d`
- `return_20d`
- `volatility_20d`
- `drawdown_20d`
- `future_return_pct_next_day`
- `future_return_pct_1w`
- `label_1d`
- `label_1w`

训练准入规则：

- 每只基金历史样本不少于 120 个交易日。
- 最近样本距离验收日不超过 10 个自然日，长期停牌或长期无净值更新的基金不进入训练。
- 标签分布极端偏斜的基金类型先进入回退，不进入通用方向模型。
- 货币基金单独输出收益型样本，不进入涨跌方向分类。

## 数据源设计

优先使用本地已验证依赖和公开 HTTP 源：

- AkShare `fund_name_em`：全基金名称和类型。
- AkShare `fund_open_fund_info_em`：开放式基金历史单位净值。
- AkShare `fund_open_fund_daily_em` / 东方财富排行接口：最新开放式基金净值和阶段收益。
- AkShare `fund_money_fund_daily_em`：货币基金收益。
- AkShare `fund_etf_hist_em`、`fund_lof_hist_em`：场内 ETF/LOF 历史行情。
- `simonlin1212/a-stock-data` 风格的东方财富直连 HTTP：作为 ETF、市场因子、行情源补强方向，避免关键数据完全依赖封装库。

## 前端设计

前端不新增复杂页面，优先补齐现有搜索和预测展示：

- 搜索结果展示基金类型和行情日期，避免用户误读旧涨跌幅。
- 预测卡片展示模型覆盖状态。
- 对 `unsupported_fund` 和 `baseline_only` 显示清晰提示，说明当前不是训练模型预测。
- 对货币基金显示收益口径提示，不展示类似股票型基金的涨跌方向承诺。

## 验收标准

- 本地基金库数量不低于 2.6 万。
- 搜索典型样例可命中：沪深300、中证500、货币、债券、QDII、纳斯达克、华夏中证A500。
- 预测接口对任意本地基金返回 200 或明确的业务错误，不因模型样本缺失崩溃。
- 覆盖报告包含：
  - total_funds
  - funds_with_quote
  - funds_with_history
  - funds_with_trainable_samples
  - model_supported_funds
  - baseline_only_funds
  - unsupported_funds
  - counts_by_fund_type
  - counts_by_quote_source
  - sample_cutoff_time
- 训练管线能产出 `all_fund_nav_daily_v1` 样本和报告。
- 现有 Go 测试、前端构建、Python 测试通过。
- 服务重启后，前端、后端、模型服务健康检查通过。

## 风险与处理

- 全量历史抓取耗时长：先提供 `max_funds`、断点复用和覆盖报告，MVP 验收时跑代表性全类型样本，之后扩大批次。
- 数据源字段变化：采集器保留字段别名解析和错误报告，不静默吞掉失败。
- 涨跌幅口径不同：按基金类型保留 quote source 和 quote date，前端展示口径，不把净值日涨跌和实时交易涨跌混为一谈。
- 模型效果不佳：MVP 阶段只把模型作为数据链路验证，不把低置信结果标成可行动信号；高级模型训练前再扩大样本和做回测。

## 实施顺序

1. 增加全基金覆盖报告和验收脚本。
2. 增加通用基金历史净值采集器。
3. 增加 `all_fund_nav_daily_v1` 样本构建。
4. 训练一个通用日频 MVP baseline/champion。
5. 后端按覆盖清单路由模型或基线。
6. 前端补齐覆盖状态和行情日期展示。
7. 重启服务，执行搜索、预测、构建、测试和健康检查。
