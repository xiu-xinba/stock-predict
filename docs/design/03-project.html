1. 问题陈述

构建一个面向金融从业者的开源基金预测平台。核心命题：在盘中实时预测基金当日收盘涨跌，而非仅仅展示当前净值估算。差异化于天天基金/蚂蚁财富等平台的关键在于：

  1. 从「展示现在」跃迁到「预测结果」
  2. 模型开源透明、预测因子可解释，附带置信区间
  3. 融合多维实时市场信号（大盘指数、行业轮动、资金流向）

目标用户画像：证券分析师、基金经理、操盘手——他们需要精确度、可解释性、实时性。最终愿景是演化成量化策略研究平台（自定因子、回测框架、策略模拟器）。

2. 方案选型

A：MVP — 单基金预测原型

Jupyter Notebook 验证 + FastAPI + Vue 3 单基金搜索预测页。工作量 M，风险中，快速验证但无法体现仪表盘魅力。

B：理想架构 — 全功能量化预测平台

一步到位：数据管道 + 多模型集成 + 实时预测引擎 + Vue 全功能仪表盘。工作量 XL，风险高，一步呈现完整产品但验证周期长。

C：渐进式两阶段 ✅ 已采纳

  • 阶段 1：Notebook 验证 + FastAPI 单基金预测 API + Vue 搜索预测页
  • 阶段 2：多基金数据管道 + 模型升级/集成 + Vue 全功能仪表盘

理由：研究先行降低风险；每阶段产出完整可展示里程碑；API 预留扩展平滑演进。

5. API 接口规范

统一响应格式

interface ApiResponse<T> {
  code: number       // 0 = 成功，非0 = 失败
  message: string    // 成功时 "success"，失败时错误描述
  data: T | null     // 成功时为数据，失败时为 null
}

已定义端点清单

#方法路径前端已对接优先级说明
1GET/api/v1/predict/{fund_code}是P0基金涨跌预测（最复杂端点）
2GET/api/v1/funds/search是P0基金搜索（模糊匹配代码或名称）
3GET/api/v1/market/indices是P0大盘指数实时数据
4GET/api/v1/market/ranking/{type}是P0基金涨跌幅排行
5POST/api/v1/watchlist/quotes是P0自选基金批量估值
6GET/api/v1/health否P0健康检查
7GET/api/v1/market/indices/{code}/chart已封装未调用P2指数走势图数据
8GET/api/v1/predict/{code}/accuracy否P1预测准确率统计
9GET/api/v1/predict/{code}/history否P1历史预测记录
10GET/api/v1/market/sectors/heatmap否P2板块热力图数据

端点详细规范

EP-1: GET /api/v1/predict/{fundCode}

最复杂的端点，响应包含嵌套的预测结果、因子列表和市场快照。

  • 路径参数：fundCode — 6 位数字（前端已验证 ^\d{6}$）
  • 响应：ApiResponse<PredictionData>
  • 前端消费字段：全部消费（prediction.* + market_snapshot.*），除 update_time
  • 后端注意：可在耗时较长（模型推理），需在 15s 内返回；前端有竞态保护

EP-2: GET /api/v1/funds/search

  • 查询参数：keyword: string, page: number(默认1), size: number(默认20)
  • 响应：ApiResponse<FundSearchData>
  • 前端消费字段：仅 items[]（fund_code, fund_name, fund_type）
  • 后端注意：需支持模糊匹配（代码或名称）

EP-3: GET /api/v1/market/indices

  • 响应：ApiResponse<MarketIndex[]>
  • 前端消费字段：name, value, change, change_pct, mini_chart_data
  • 后端注意：3 个请求同时到达（与 ranking 并行），需确保数据时间一致性；30 秒轮询频率，需考虑缓存策略；mini_chart_data 需返回足够数据点（建议 20-30 个）

响应示例:

{
  "code": 0,
  "message": "success",
  "data": [
    {
      "code": "000001",
      "name": "上证指数",
      "value": 3350.12,
      "change": 15.34,
      "change_pct": 0.46,
      "mini_chart_data": [3320, 3325, 3330, 3340, 3335, 3350],
      "update_time": "2026-05-18T10:30:00+08:00"
    }
  ]
}

EP-4: GET /api/v1/market/ranking/{type}

  • 路径参数：type — gainers 或 losers
  • 查询参数：size: number（前端传 5）
  • 响应：ApiResponse<FundRankingItem[]>
  • 前端消费字段：rank, fund_code, fund_name, fund_type, change_pct

EP-5: POST /api/v1/watchlist/quotes

  • 请求体：{ "codes": ["000001", "110011"] }
  • 响应：ApiResponse<WatchlistItem[]>
  • 前端消费字段：仅 fund_code, estimated_nav, change_pct, direction
  • 后端注意：批量查询，codes 数组可能包含 1-50 个基金代码；空列表时前端不调用

EP-6: GET /api/v1/health

  • 响应：{ "status": "ok", "model_loaded": false }

前端隐式约定（后端必须了解）

约定说明
CSRF Cookie前端从 document.cookie 读取 csrftoken，后端需设置此 cookie
X-CSRFToken Header前端每个请求携带此 header，后端需验证
withCredentials前端全局开启，后端 CORS 需 allow_credentials=True
请求去重前端自动取消相同 GET 请求，后端无需特殊处理
超时 15s前端 15 秒超时，预测接口需在此时间内返回
5xx 脱敏前端将 5xx 错误转为通用提示，开发环境保留详情
snake_case前后端统一使用 snake_case，无需 camelCase 转换

数据库建议

  1. Phase 1 使用 SQLite：轻量级，无需额外服务
  2. fund_search 可用内存列表：主计划已规划内置基金列表
  3. mini_chart_data 存 JSON：SQLite 支持 JSON 扩展，或直接 TEXT 存储
  4. Phase 2 迁移 PostgreSQL：支持并发、JSONB、全文搜索

7. 前后端交互流程

流程 1: 市场行情加载

用户访问 /market
    ↓
MarketView.onMounted()
    ↓
marketStore.startRefresh() → fetchMarketData()
    ↓
Promise.all([
  fetchMarketIndices(),           → GET /api/v1/market/indices
  fetchFundRanking('gainers', 5), → GET /api/v1/market/ranking/gainers?size=5
  fetchFundRanking('losers', 5),  → GET /api/v1/market/ranking/losers?size=5
])
    ↓
3 个请求并行发出，后端需支持并发
    ↓
setInterval(30s) 持续轮询

流程 2: 基金搜索与预测

用户输入关键词 → 300ms 防抖 → GET /funds/search?keyword=xxx
    ↓
用户选择基金 → 前端验证 /^\d{6}$/ → GET /predict/{code}
    ↓
竞态保护: predictSeq 确保旧请求不覆盖新结果

流程 3: 自选列表行情刷新

用户访问 /watchlist
    ↓
watchlistStore.refreshQuotes()
    ↓
POST /watchlist/quotes  body: { "codes": ["000001", ...] }
    ↓
响应: quotes 数组 → 更新本地 items 的 estimated_nav, change_pct, direction
    ↓
setInterval(30s) 持续轮询

9. 风险评估

高风险

#风险影响缓解措施
R1东方财富 API 反爬数据采集失败，整个平台无数据实现请求限速、User-Agent 伪装、多数据源备选
R2模型推理超时/predict 接口超过 15s 前端超时预计算热门基金、模型缓存、异步推理队列
R3实时数据延迟行情数据不及时，影响预测准确性服务端缓存 + TTL 30s、WebSocket 推送（Phase 2）

中风险

#风险影响缓解措施
R4CORS 跨域问题前端无法调用后端 API后端配置 CORSMiddleware(allow_methods=["GET", "POST"])
R5CSRF Token 不匹配POST 请求被拒绝Phase 1 可禁用 CSRF（仅 API 无认证）
R6SQLite 并发写入多请求同时写入时锁定Phase 1 以读为主；Phase 2 迁移 PostgreSQL
R7基金列表数据源内置列表需定期更新首次从东方财富采集全量列表并缓存

低风险

#风险影响缓解措施
R8模型文件体积Git 仓库膨胀Git LFS 或单独存储
R9环境变量泄露安全风险.env 加入 .gitignore
R10snake_case/camelCase 不一致字段映射错误前后端已统一 snake_case，无需转换
