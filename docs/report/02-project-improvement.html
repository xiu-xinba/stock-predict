一、前端

1.1 现有问题清单

零测试覆盖 P0 严重

项目无 vitest/jest 依赖，所有前端逻辑（Store/Composable/工具函数）无任何测试保护，回归风险极高。

  • 3个 Pinia Store（竞态防护、localStorage持久化、请求去重）均无测试
  • 3个 Composable（useECharts/useFundSearch/useTheme）无测试
  • API 层拦截器逻辑（去重/取消/CSRF/错误处理）无测试
搜索结果无分页UI P1 高

后端支持 page/size 参数，前端 Store 支持 page 传入，但搜索结果无分页 UI，用户只能看到第一页（最多20条）。

冗余组件未清理 P2 中

3个组件未被任何页面引用，属于死代码：

  • PredictionChart.vue — 功能已被 PredictionCard 内嵌图表替代
  • IndexCard.vue — 已被 MarketSidebar 替代
  • MarketSection.vue — 已被 MarketSidebar 替代
大组件未拆分 P2 中

两个核心组件代码量过大，维护困难：

  • PredictionCard.vue — 约630行，承担方向徽章/多周期预测/因子列表/因子图表/可靠性/数据质量等过多职责
  • FundSearch.vue — 约600行，搜索逻辑/历史记录/筛选面板/下拉建议全部耦合

1.2 改进建议

引入 Vitest 单元测试 P0

建立前端测试基础设施，优先覆盖核心逻辑。

  • 安装 vitest + @vue/test-utils + @pinia/testing
  • 优先测试：3个 Store（竞态防护/去重/持久化）、3个 Composable、API 拦截器
  • 目标覆盖率：核心逻辑 > 80%
预期效果：回归缺陷率降低 60%+，重构信心大幅提升
搜索分页UI P1

在 FundSearch 组件底部添加分页控件，支持翻页和每页条数切换。

预期效果：用户可浏览全部搜索结果，不再受限于20条
拆分大组件 P2

将 PredictionCard 和 FundSearch 拆分为更小的子组件：

  • PredictionCard → HorizonGrid + FactorList + MetricsGrid + ReliabilityWarning
  • FundSearch → SearchInput + SearchDropdown + FilterPanel + SearchHistory
预期效果：单组件行数降至 200 行以内，可维护性显著提升
统一 ECharts 管理方式 P2

MarketSidebar 手动管理 ECharts 实例，与其他组件使用 useECharts 不一致。重构为统一使用 useECharts。

添加全局错误边界 P2

无 Vue errorHandler 或 onErrorCaptured，组件渲染异常可能导致白屏。添加全局错误处理和降级 UI。

1.3 优化方向

性能优化 P2
  • 预压缩：添加 vite-plugin-compression，gzip 预压缩静态资源
  • Web Vitals：集成 web-vitals 库，监控 LCP/FID/CLS
  • 离线检测：navigator.onLine 监听，网络断开时提示用户
  • deep: true 清理：PredictionChart/IndexCard 中不必要的 deep watcher 移除
代码质量 P2
  • console.error 残留：替换为可配置日志服务
  • 硬编码值提取：API 超时 15000ms、刷新间隔 30000ms、图表 Y 轴 max: 0.4 等提取为配置常量
  • CSRF 逻辑评估：当前无用户认证体系，CSRF Token 读取逻辑可能多余
  • 刷新机制统一：WatchlistView 自行管理 setInterval，MarketView 使用 Store 引用计数，两种模式应统一

二、后端

2.1 现有问题清单

数据层完全虚构 P0 严重

所有数据均为硬编码或伪随机生成，无任何真实数据源接入：

  • 基金列表：30条硬编码种子数据（seedFunds()）
  • 基金净值/收益：基于 FNV 哈希生成（enrichFund()）
  • 市场指数：8个指数 + 确定性伪随机涨跌（deterministicChange()）
  • 迷你图表：60点正弦波模拟（miniChart()）
  • 预测因子：5个固定因子 + 固定权重
  • MeetsAccuracyTarget 始终 false
  • IsActionable 基线始终 false
无数据持久化 P0 严重

纯内存存储（sync.RWMutex + map），进程重启即丢失。无写入方法（无 AddFund/UpdateFund/DeleteFund），数据不可变。

syncFunds 端点为空壳 P0 高

POST /api/v1/funds/sync 仅返回基金数量，不触发任何外部数据拉取或更新操作。

预测模型已拆分为独立项目 P0 高

当前主项目仅保留预测入口占位，模型训练与推理不再由该仓库维护。

无用户认证体系 P1 高

仅 X-Admin-Token 简单校验用于 sync 接口，无用户注册/登录/JWT/Session 机制。

无实时推送能力 P1 中

盘中数据无法实时更新，前端仅30秒轮询。无 WebSocket/SSE 支持。

测试覆盖不足 P2 中

MarketService、ModelClient、middleware、config、store 均无独立测试。整体覆盖率约 40-50%。

2.2 改进建议

接入真实数据源 P0

利用模型训练模块已有的 akshare/mootdx 采集管道，为后端提供真实数据。

  • 方案A：Go 后端直接调用 akshare Python 服务（通过 HTTP 微服务）
  • 方案B：Go 后端实现独立的数据采集（调用东方财富/Tushare API）
  • 方案C：Python 定时采集 → 写入数据库 → Go 后端读取
  • 推荐方案C：解耦采集与查询，Go 后端只负责读取
预期效果：行情数据从虚构变为真实，平台可用性从演示级提升至生产级
引入数据库持久化 P0

用 SQLite（Phase 1）或 PostgreSQL（Phase 2）替代内存存储。

  • 实现 FundRepository 接口的数据库版本
  • 添加 AddFund/UpdateFund/DeleteFund 写入方法
  • 实现 syncFunds 真实同步逻辑
  • 添加数据库迁移机制
预期效果：数据重启不丢失，支持增量更新，为用户体系奠定基础
保留预测入口占位 P0

预测模型训练与推理拆分为独立项目；当前主项目只维护入口、路由和占位文案。

  • 有效代码的预测 API 返回 HTTP 501。
  • 前端预测页展示独立项目占位。
预期效果：预测结果从基线公式变为真实模型推理，预测准确率显著提升
添加用户认证体系 P1

引入 JWT 认证，支持用户注册/登录/自选列表云端同步。

  • Phase 1：JWT 签发 + 中间件校验 + 用户表
  • Phase 2：自选列表双写（localStorage + 后端）
  • Phase 3：OAuth2 第三方登录（微信/GitHub）
添加 WebSocket 实时推送 P1

替代30秒轮询，实现盘中数据实时更新。

  • Go 后端使用 gorilla/websocket 或 nhooyr.io/websocket
  • 推送内容：指数行情、基金估值、预测结果更新
  • 前端使用原生 WebSocket + 自动重连 + 心跳检测
预期效果：数据延迟从 30s 降至 < 1s，盘中体验质变

2.3 优化方向

性能优化 P1
  • 搜索索引：当前全量线性扫描，数据量大时性能堪忧。引入 Bloom Filter 或倒排索引
  • 缓存层：添加 Redis 或内存缓存（TTL 30s），减少重复查询
  • API 限流：添加 rate limiting 中间件，防止滥用
  • 模型服务熔断：添加 circuit breaker，模型服务不可用时快速降级而非超时等待
安全加固 P1
  • HTTPS：所有服务启用 TLS
  • CORS 精细化：生产环境限制具体域名，非 *
  • 输入校验增强：所有端点参数添加边界校验
  • 敏感信息保护：ADMIN_TOKEN 等环境变量加密存储
代码质量 P2
  • 补充测试：MarketService、middleware、config、store 独立测试
  • 基准测试：添加 Go benchmark 测试关键路径性能
  • API 文档：生成 OpenAPI/Swagger 规范
  • 修正文档：根 README 将"Go 标准库 net/http"更正为"Gin 框架"

三、预测模型

3.1 现有问题清单

sklearn 模型无 ONNX 导出 P1 高

仅 LightGBM 路径支持 ONNX 导出。baseline 和 tournament 使用的 sklearn 模型（HistGBDT/RandomForest/ExtraTrees 等）仅导出 joblib 格式，无法跨语言部署。

模型格式过渡期矛盾 P1 中

文档重点描述 ONNX 合约，但当前 champion 模型均为 joblib 格式。ONNX 导出代码存在但未在生产链路启用。文档与实际不一致。

高级 MLOps 功能未实现 P2 中

配置文件中已规划但代码未实现的功能：

  • 流式漂移检测（ADWIN/Page-Hinkley/KSWIN）— 需集成 river
  • IC 衰减 / Rank IC 衰减监控
  • 模型校准（ECE 指标）
  • MLflow 模型注册集成
  • 特征存储（Feast）集成
  • 编排器（Airflow/Prefect/Dagster）集成
推理服务依赖预计算 CSV P2 中

预测模型训练与推理已拆分到独立项目，当前主项目不维护实时特征计算。

3.2 改进建议

sklearn 模型 ONNX 导出 P1

引入 skl2onnx 库，为 baseline/tournament 的 sklearn 模型实现 ONNX 导出。

  • 安装 skl2onnx + onnxruntime
  • 为 HistGBDT、RandomForest、ExtraTrees、GradientBoosting、LogisticRegression 各实现导出
  • 统一 ONNX 合约：输入 float_input，输出 label + probabilities
  • 导出后自动验证后端合约
预期效果：所有模型统一为 ONNX 格式，Go 后端可直接加载推理，消除 Python 服务依赖
实时特征计算 P1

在推理服务中实现实时特征计算，替代从 CSV 读取。

  • 方案A：推理服务内嵌特征计算模块，从数据库读取原始行情数据
  • 方案B：独立特征服务，推理服务调用特征服务获取特征向量
  • 推荐方案A：减少服务间调用延迟
预期效果：任意基金/任意日期均可即时推理，不再依赖预计算 CSV
渐进式实现 MLOps 功能 P2

按优先级逐步实现高级 MLOps 功能：

  • Phase 1：IC 衰减监控 + 模型校准（ECE）— 代码量小，价值高
  • Phase 2：流式漂移检测（ADWIN）— 需引入 river 依赖
  • Phase 3：MLflow 集成 — 实验追踪 + 模型注册
  • Phase 4：编排器集成 — 自动化训练/评估/部署流水线

3.3 优化方向

模型质量提升 P1
  • 特征扩展：当前62个特征，可增加舆情因子（新闻情感/社交媒体）、宏观因子（利率/CPI/PMI）、技术因子（MACD/RSI/布林带）
  • 模型升级：从传统 ML 向深度学习演进（LSTM/Transformer），需充足数据量
  • 集成策略优化：从 Voting 升级为 Stacking，引入元学习器
  • 校准优化：实现 Platt Scaling / Isotonic Regression，提升置信度可靠性
训练流水线优化 P2
  • 数据版本化：DVC 或 LakeFS 管理训练数据版本
  • 特征存储：Feast 管理特征版本和在线/离线特征服务
  • 超参优化：Optuna/Ray Tune 替代手动调参
  • 分布式训练：Ray 或 Spark 支持大规模数据训练

四、部署运维

4.1 现有问题清单

无容器化配置 P0 高

项目无 Dockerfile、docker-compose.yml，所有服务需手动启动和配置，环境一致性无法保障。

无 CI/CD 流水线 P1 高

无 GitHub Actions / GitLab CI 等配置，代码推送后无自动测试/构建/部署流程。

无监控告警 P1 中

无 Prometheus/Grafana/AlertManager 等监控，服务异常无法及时发现。无日志聚合（ELK/Loki）。

4.2 改进建议

Docker 容器化 P0

为三个服务分别创建 Dockerfile，编写 docker-compose.yml 一键部署。

  • 前端：多阶段构建（node:20-alpine 构建 → nginx:alpine 运行）
  • 后端：golang:1.23-alpine 构建 → scratch/alpine 运行
  • 模型服务：python:3.12-slim + 模型文件挂载
  • docker-compose.yml：3个服务 + 网络 + 卷 + 环境变量
预期效果：一键 docker compose up 启动全栈，环境一致性 100%
CI/CD 流水线 P1

建立 GitHub Actions 自动化流水线。

  • 前端：lint → typecheck → test → build → 部署到 Vercel/Cloudflare Pages
  • 后端：lint → test → build → 推送 Docker 镜像 → 部署到 Fly.io/Render
  • 模型训练：lint → test → 训练 → 评估 → 模型注册
监控告警体系 P1

  • Prometheus + Grafana：服务指标（QPS/延迟/错误率）+ 模型指标（准确率/漂移分数）
  • AlertManager：服务宕机/模型准确率下降/漂移超阈值告警
  • Loki + Promtail：日志聚合和查询

4.3 优化方向

生产化部署 P1
  • HTTPS：Let's Encrypt + Nginx 反向代理自动证书
  • CDN：前端静态资源通过 Cloudflare/CloudFront 分发
  • 数据库：SQLite → PostgreSQL，支持并发和 JSONB
  • 备份：数据库定时备份 + 模型文件版本化存储
高可用架构 P3
  • 负载均衡：Nginx/HAProxy 多实例负载
  • 模型服务副本：多副本推理服务 + 请求分发
  • 数据库主从：PostgreSQL 主从复制
  • 蓝绿部署：零停机更新
