3. 前端审查

生产构建当前不可交付

P0

frontend/src/components/SearchOverlay.vue:125

`npm run build` 在 TypeScript 编译阶段失败，原因是从 `vue` 导入的 `ref` 未使用。这个问题本身很小，但影响级别很高：只要构建失败，前端无法生成可发布产物，任何后续 UI 或功能验收都不具备商业交付意义。

治理建议：P0 阶段删除未使用导入，并将 `npm run build` 纳入 CI 必过项。构建错误必须视为发布阻断，而不是开发警告。

ESLint 环境配置与代码规则不匹配

P1

frontend/eslint.config.js

`npm run lint` 报出大量 `window`、`document`、`setTimeout`、`HTMLElement`、`KeyboardEvent` 未定义错误。项目实际是浏览器 SPA，但 ESLint flat config 未声明 browser globals，导致规则噪音淹没真实问题。

治理建议：为前端 ESLint 配置浏览器全局变量，同时保留 `no-empty`、`no-unused-vars`、`no-explicit-any` 等质量规则。第一阶段目标不是降低规则，而是让规则能准确表达项目运行环境。

CSRF 前后端实现存在契约冲突风险

P1

backend-go/internal/api/middleware.go:223
frontend/src/api/index.ts:73

后端设置 `csrf_token` Cookie 时启用了 HttpOnly；前端却通过 `document.cookie` 读取该 Cookie 并写入 `X-CSRF-Token`。HttpOnly Cookie 不能被 JavaScript 读取，因此在启用凭据和真实 POST 流程时，前端可能无法提供 CSRF Header。

治理建议：明确 CSRF 方案二选一。方案 A：后端通过非 HttpOnly CSRF Cookie 或响应头提供可读令牌，前端读取并提交 Header。方案 B：使用 BFF/session 方案，让前端不直接读取令牌。当前实现必须通过集成测试覆盖。

类型安全已有基础，但缺少契约自动校验

P2

frontend/src/api/*.ts
docs/api/openapi.yaml

前端 API 层已有 `ApiResponse` 和领域类型，OpenAPI 文档也覆盖主要接口。但 TypeScript 类型、Go DTO 和 OpenAPI 之间仍是手动同步，缺少生成或差异检测机制。随着股票、基金、预测三套模型继续演化，字段漂移会成为高频风险。

治理建议：以 OpenAPI 为契约源，建立至少一条自动校验链路：后端 handler 测试对齐 OpenAPI 示例，前端类型由 OpenAPI 生成或用 contract tests 验证关键接口字段。

部分组件承担过多职责

frontend/src/views/WatchlistView.vue:584 lines

GOOD

安全策略需要从“已有实现”升级为“可验收策略”

P1

backend-go/internal/api/middleware.go

后端已有安全头、CORS、CSRF、限流、管理员令牌和请求体大小限制，这是良好基础。但生产化还缺少可执行验收：CORS 禁止通配符与 credentials 组合、CSRF 令牌流向、管理员令牌强度、反向代理下 ClientIP 策略、限流粒度都需要自动测试或部署文档约束。

治理建议：建立安全验收测试集，覆盖 OPTIONS、GET 令牌下发、POST CSRF 成功/失败、管理员接口认证、生产环境 CORS 配置和 429 行为。

服务层文件规模偏大，后续演进成本会上升

P2

backend-go/internal/service/prediction_service.go:561 lines
backend-go/internal/service/stock_sync.go:545 lines
backend-go/internal/service/stock_detail_service.go:401 lines

预测服务同时承担基线估计、模型服务调用、数据质量、周期组合和降级策略；股票同步同时承担 HTTP 获取、解析、清洗、拼音和合并。它们已经是稳定业务核心，后续新增模型或数据源时应拆成更细边界。

治理建议：先通过接口测试固定行为，再拆出 `prediction/baseline`、`prediction/client`、`prediction/quality`、`stocksource/eastmoney`、`stocksource/normalize` 等内部包或文件组。

外部数据源适配层需要统一超时、重试和错误分类

P2

基金、股票、行情和模型服务都依赖外部 HTTP 服务。当前已有 URL 白名单和模型客户端熔断思路，但数据源调用策略散落在不同 service 中。商业运营环境中，外部数据源抖动是常态，需要统一错误分类、重试预算、降级策略和监控指标。

治理建议：抽象 `DataProvider` 和 `ProviderError`，记录来源、请求耗时、失败原因、重试次数，并将降级行为暴露给前端的数据质量字段。

6. 架构优化蓝图

推荐目标不是重建项目，而是在当前三段式结构上建立“边界更清晰、契约可验证、变更可追踪”的商业级工程框架。

目标目录结构

目录 定位 治理要求
frontend/src/api 唯一 HTTP 接入层 从 OpenAPI 或契约测试保障类型一致；禁止组件直接拼接口路径。
frontend/src/components 展示组件与小型交互组件 大型业务组件拆出 composables；组件 props 和 emits 类型化。
frontend/src/stores 领域状态和缓存策略 明确竞态防护、错误状态、刷新节流和持久化边界。
backend-go/internal/api HTTP handler、中间件、响应格式 只做参数、认证和响应，不承载业务决策。
backend-go/internal/service 业务用例层 按领域拆分预测、搜索、行情、同步；大文件逐步拆为内部协作者。
backend-go/internal/store 持久化和搜索索引 统一接口、事务边界和测试 fixture；避免业务规则进入存储层。
model-training/src/fund_model_training 训练、特征、评估、服务 特征工程函数可测试、模型发布门禁文档化、数据采集适配器隔离。
docs 架构、API、部署、维护、治理报告 报告与执行计划分离；OpenAPI 作为接口契约源。

接口契约治理

  • 以 docs/api/openapi.yaml 作为外部接口契约源，所有新增接口必须先更新或生成契约。
  • 前端 API 层只能依赖 API_ROUTES 和类型化方法，组件不得直接引用硬编码 URL。
  • 后端 handler 测试覆盖 OpenAPI 中的关键响应结构，包括错误响应。
  • 预测结果字段必须保留数据质量、信号状态和模型覆盖说明，避免给用户制造虚假确定性。
