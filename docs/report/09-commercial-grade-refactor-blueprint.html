建议治理阶段

当前状态判断

项目已经完成从单体原型向三段式系统的初步演进，具备 Vue 前端、Go API、Python 模型训练与 OpenAPI 文档。它不是缺少结构，而是需要把质量门禁、契约治理和生产化边界补齐。

商业化主要风险

当前无法直接进入商业运营。前端生产构建失败，模型训练测试失败，lint 未形成有效门禁，接口契约仍依赖人工同步，模型服务与数据质量策略需要更明确的上线门槛。

推荐路径

采用分阶段治理：先清除 P0 阻塞并建立验证基线，再做前后端契约和目录边界治理，最后补齐部署、监控、版本策略和维护手册。

本轮修复记录： 2026-05-31 已完成 P0/P1 可验证项处理：前端生产构建恢复、ESLint error 清零、模型训练 pytest 全量通过，并新增部署、维护和模型发布手册。随后补齐 P2/P3 的可落地项：新增 `/api/v1/metrics` 运行指标接口、API 契约校验脚本、商业化发布总门禁脚本，并将前端 ECharts vendor chunk 纳入明确构建预算。

前端审查

Vue 3 + TypeScript + Pinia + Vite
P0
生产构建当前失败，商业交付链路被阻断
FE-001

证据：npm run build 在 SearchOverlay.vue 的 ref 导入处失败。该问题修复成本很低，但它说明前端构建没有被持续作为合并门禁执行。

建议：将 npm run build、npm run lint、npm run test:run 纳入 P0 门禁。任何架构调整前先保证主干可构建。

P1
ESLint 配置未正确声明浏览器环境，导致大量噪声 error
FE-002

证据：document、window、KeyboardEvent、HTMLElement、setTimeout 等浏览器全局变量被标记为 no-undef。这不是业务缺陷，但会让 lint 报告失去信号密度。

建议：在 frontend/eslint.config.js 中引入浏览器 globals 或关闭 TypeScript 项目的核心 no-undef 冲突规则，再处理真实 lint 问题。

P1
部分组件过大，交互状态、数据读取与展示逻辑混杂
FE-003

当前较大的前端文件包括 SearchOverlay.vue、WatchlistView.vue、App.vue、PredictionCard.vue、MarketDock.vue。这些文件并非不可维护，但已经出现多个职责并存。

建议：按领域拆分为 features/search、features/watchlist、features/market、features/prediction、shared/ui。每次拆分必须保持测试通过，不做纯搬家式大爆炸重排。

P2
API 类型契约已经有基础，但仍缺少自动一致性校验
FE-004

前端 src/types 与后端 dto、OpenAPI 文档整体方向一致，但当前仍依赖人工同步。字段采用 snake_case 是合理选择，因为能减少转换层复杂度。

建议：建立 OpenAPI 为契约源的检查流程。短期先编写契约对照清单，中期生成 TypeScript client/types，长期将 OpenAPI diff 纳入 CI。

后端审查

Go + Gin + Store/Service/API 分层
健康
后端基础质量门禁当前通过，可以作为重构锚点
BE-001

go test ./... 和 go vet ./... 均通过。API、config、service、store 已经具备测试基础，说明后端可以采用小步重构而不是推倒重来。

P1
中间件和安全策略已有雏形，但生产策略仍需外部化和可观测化
BE-002

当前已经有恢复、请求 ID、日志、安全响应头、CORS、CSRF、gzip、请求体限制和 IP 限流。风险主要不在“没有安全功能”，而在生产参数固定、指标不可观测、策略难以按环境调整。

建议：把限流阈值、CSRF cookie 名称、允许方法、gzip 策略、请求体上限纳入配置，并输出最小运行指标：请求总数、错误数、延迟、限流命中数、外部数据源失败数。

P2
服务层领域边界清晰度还可以继续提升
BE-003

后端已有 api、service、store、dto 分层，但 prediction_service.go、stock_sync.go、stock_detail_service.go 等文件较大，且外部数据源解析、领域计算、缓存、降级策略有继续分离空间。

建议：优先拆出外部 provider、领域 assembler、cache policy 和 fallback policy。每个拆分配套测试，保持 Handler 和 DTO 对外契约不变。

P2
搜索索引与持久化策略适合 MVP，但商业运营需要数据生命周期设计
BE-004

SQLite FTS5 内存索引配合 JSON 持久化适合开发期和轻量部署。商业运营需要明确数据刷新频率、失败重试、数据版本、回滚、冷启动耗时与索引重建策略。

建议：短期保持现状，补充数据同步操作手册和失败恢复流程；中期评估持久化 SQLite 或外部数据库；长期引入任务调度和数据版本审计。

模型训练审查

Python training pipeline + model serving
P0
模型训练测试存在单点失败，阻断全量质量门禁
ML-001

证据：python -m pytest 失败于 tests/test_build_samples.py::BuildSamplesTests::test_build_daily_weekly_samples_with_market_features。根因位于 features.py 的 _drawdown_20d，groupby apply 返回结构在当前 pandas 版本下变成多列 DataFrame。

建议：先用测试驱动修复 drawdown 特征输出为与输入索引对齐的一维 Series，并加一个多基金样本用例，防止回归。

P1
特征工程需要更明确的 schema 约束和版本策略
ML-002

当前已经有 schema.py、validate_contract.py、多组 pipeline 和模型注册测试，这是很好的基础。商业化风险在于特征集演化、训练数据窗口、缺失值填充和预测服务输入契约需要严格版本化。

建议：每个 feature_set 固化输入列、派生列、标签列、目标 horizon 和数据时效要求。模型产物必须绑定 feature_set 版本和训练数据摘要。

P2
模型上线需要区分“可运行”和“可行动信号”
ML-003

项目文档已强调 signal_status、置信度、数据质量和降级策略。后续商业化应继续强化这个方向：模型服务可用不等于预测可用于决策参考。

建议：上线门禁包含离线指标、shadow evaluation、漂移报告、数据覆盖率、置信区间和 fallback 标记。前端必须诚实展示低置信、未覆盖和基线估计。

架构优化蓝图

保持现有三段式系统，做边界治理

推荐目标目录方向

  • frontend/src/features/*：按业务域组织页面、组件、store 和 API 适配。
  • frontend/src/shared/*：通用 UI、工具、图表、错误处理、主题能力。
  • backend-go/internal/domain/*：沉淀基金、股票、预测、市场等领域模型和策略。
  • backend-go/internal/provider/*：隔离东方财富、腾讯、模型服务等外部依赖。
  • 预测模型训练与推理：作为独立项目治理，与当前主项目通过后续合同对接。
  • docs/operations/*：部署、同步、回滚、监控、故障处理手册。

不建议立刻做的事

  • 不建议一次性重命名所有目录，这会制造大量低价值 diff。
  • 不建议先引入复杂微服务，当前复杂度更适合清晰模块化单体加独立模型服务。
  • 不建议先替换 UI 框架或图表库，现有栈足够支撑商业化 MVP。
  • 不建议在测试失败时继续大规模重构，先恢复可验证基线。
架构原则：后续优化应围绕“模块边界可理解、接口契约可验证、数据质量可追踪、生产行为可观测”展开。商业级不是文件夹更漂亮，而是每个变更都有保护网、每个失败都有恢复路径。

分阶段治理路线图

按风险和依赖顺序推进

P0：恢复可交付基线

修复前端构建失败、模型训练失败测试、ESLint 配置噪声。目标是所有基础命令可通过，后续重构有保护网。

  • 修复 SearchOverlay.vue 未使用导入。
  • 用测试驱动修复 _drawdown_20d 输出对齐问题。
  • 调整 ESLint browser globals，并清理真实 lint errors。

P1：建立质量门禁和契约基线

把可运行变成可持续开发。统一验证命令、接口契约清单、错误处理规范和最小文档。

  • 新增根级检查脚本或文档化命令矩阵。
  • 前后端 API routes、OpenAPI、DTO、TS 类型做一次契约对照。
  • 补充关键业务流测试：搜索、自选、预测、模型 fallback。

P2：模块化重构

在测试保护下拆分大文件，梳理前端 features/shared，后端 provider/domain，模型 feature_set 边界。

  • 前端优先拆搜索、自选、行情和预测域。
  • 后端优先拆外部 provider、fallback policy 和 cache policy。
  • 模型训练优先拆特征工程与数据 contract。

P3：生产化运营

补齐部署、监控、运维和模型发布流程，让系统具备商业运营所需的可观测性和恢复能力。

  • 生产部署指南、回滚手册、数据同步手册。
  • 日志结构化、指标采集、外部 API 失败告警。
  • 模型注册、shadow evaluation、漂移监控和发布审批。

  • 每次重构都有测试或快照保护，避免纯目录搬迁造成隐性回归。
